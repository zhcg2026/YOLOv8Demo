# 巡查速度与轨迹修复说明

## 1. 修复目标
解决以下移动端巡查问题：
1. 结束巡查后菜单页需要返回两次才能退出。
2. 速度显示长期为 `0.0 km/h`，且偶发极端异常值。
3. 短时巡查轨迹里程长期为 `0 km`。

## 2. 涉及类与方法
- `InspectionActivity.finishInspection()`
  - 调整页面回退栈，避免菜单页重复入栈。
- `InspectionActivity` 位置回调内速度计算逻辑
  - 增加精度过滤、时间窗限制、最小位移限制、最大速度限制和滑动平滑。
- `InspectionActivity.smoothSpeed(speed: Float): Float`
  - 新增速度平滑方法，降低抖动导致的瞬时峰值。
- `TrackRecorder.recordPoint(latitude: Double, longitude: Double)`
  - 调整采样策略：首点立即记录，后续按时间+位移联合过滤。
- `TrackRecorder.stopAndUpload()`
  - 累计里程时增加漂移段与异常跳点过滤。
- `TrackRecorder.calcSegmentSpeedKmh(first: TrackPointData, second: TrackPointData, segmentKm: Double)`
  - 新增轨迹段速度计算方法，用于剔除不合理轨迹段。

## 3. 关键逻辑说明
### 3.1 页面返回栈
- 结束巡查时使用 `FLAG_ACTIVITY_CLEAR_TOP | FLAG_ACTIVITY_SINGLE_TOP` 返回菜单页。
- 作用：复用已存在的菜单页实例，避免重复创建导致“双返回”。

### 3.2 速度显示
- GPS速度仅在定位精度满足条件时参与计算。
- 差分速度仅在合理时间窗和最小位移条件满足时计算。
- 对速度设置上限阈值，过滤明显异常值。
- 通过 3 点滑动窗口平滑输出，降低瞬时抖动。

### 3.3 轨迹里程
- 采样周期从 10 秒缩短为 3 秒，提升短时巡查可用性。
- 首次定位立即落点，避免短任务只有结束时无有效轨迹点。
- 小位移（漂移）和异常高速度段不计入里程，减少统计偏差。

## 4. 兼容性说明
- 未修改接口协议字段。
- 未调整服务端端口和现有配置项。
- 修复以客户端逻辑增强为主，不影响历史轨迹数据读取。

## 5. 建议验证用例
1. 开始巡查后 1~2 分钟低速步行，速度应在合理区间波动，不应出现极端值。
2. 巡查结束返回菜单后，按返回键应直接退出应用（不再出现两层菜单）。
3. 进行 2~5 分钟短时巡查，轨迹里程应有稳定非零结果，且不出现明显离谱跳变。

## 6. 二次修复补充（针对复测反馈）
### 6.1 速度仍为 0 的修正
- 放宽速度计算门槛：
  - 精度门槛从 `35m` 放宽为 `80m`；
  - 差分速度最小位移从 `2m` 放宽到 `0.8m`；
  - 时间窗从 `1.5~15s` 放宽到 `0.8~30s`。
- 差分速度改为“回调到达时间”计算，避免部分设备 `location.time` 异常导致速度归零。
- 平滑逻辑改为优先使用非零速度样本均值，减少被零值稀释。

### 6.2 里程偏小的修正
- `TrackRecorder` 的漂移过滤阈值从 `3m` 降为 `0.8m`，低速步行段可被计入里程。

### 6.3 关键帧不上报/打不开的修正
- `VideoRecorder` 增加“编码成功时也保留关键帧快照”的降级保障，MP4 失败仍可回退 JPEG 上报。
- 增加 MP4 大小限制（超过阈值自动降级关键帧图），减少请求体过大导致上报失败。
- 后台页面 `showVideo()` 改为按实际媒体类型动态展示：
  - `video/*` 用 `<video>` 播放；
  - 非视频（如 JPEG）用 `<img>` 显示。
